# PACK Material Station 打包部署说明

## 一、打包过程

### 1. 前端打包

```bash
# 进入前端目录
cd c:\Users\95403\Desktop\PACK_Onlie

# 安装依赖（如果还没安装）
npm install

# 构建生产版本
npm run build
```

**构建产物位置：** `dist/`

### 2. 后端打包

```bash
cd c:\Users\95403\Desktop\PACK_Onlie\backend\MesScanner.Backend

# 清理旧构建
dotnet clean

# 发布自包含单文件exe（win-x64）
dotnet publish -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true
```

**构建产物位置：** `bin\Release\net8.0\win-x64\publish\`

### 3. 部署目录结构

```
deploy/PACK_MaterialStation/
├── backend/
│   ├── MesScanner.Backend.exe    # 后端主程序（约95MB）
│   ├── wwwroot/                  # 前端静态文件（需手动复制）
│   ├── Config/                   # 配置目录
│   └── appsettings.json
├── frontend/                     # 前端原始构建（备用）
├── Config/
│   └── app_config.json
├── BarTender/
└── 启动说明.txt
```

### 4. 部署步骤

1. 将整个 `deploy/PACK_MaterialStation/` 文件夹拷贝到目标电脑
2. 确保 `backend/wwwroot/` 目录下有前端文件（index.html 和 assets/）
3. 运行 `backend\MesScanner.Backend.exe`
4. 访问 `http://localhost:5000`

## 二、修改内容

### 1. Program.cs 修改

**问题：** 部署版本启动后访问根路径只显示 "PACK Material Station Backend Running..."，无法显示前端页面。

**原因：** 源代码版本的 Program.cs 有一个 `MapGet("/", ...)` 映射，覆盖了静态文件服务。

**修复：**

```csharp
// 添加 WebRoot 配置
var builder = WebApplication.CreateBuilder(args);
builder.WebHost.UseWebRoot("wwwroot");  // <-- 添加这行

// 确保静态文件中间件在 API 映射之前
app.UseCors("AllowAll");
app.UseDefaultFiles(new DefaultFilesOptions { DefaultFileNames = new[] { "index.html" } });
app.UseStaticFiles();

// 删除或注释掉这个会覆盖静态文件的映射
// app.MapGet("/", () => "PACK Material Station Backend Running...");
// app.MapGet("/", () => Results.Redirect("/index.html"));
```

### 2. wwwroot 目录

**问题：** 静态文件没有自动包含在发布包中。

**解决：** 需要手动创建 wwwroot 目录并复制前端文件：

```bash
mkdir -p backend/wwwroot
cp -r dist/* backend/wwwroot/
```

## 三、之前出问题的主要原因

### 问题1：前端空白页

| 原因 | 说明 |
|------|------|
| `MapGet("/")` 覆盖静态文件 | 源代码中根路径被 API 映射，导致 `/` 返回文本而非 index.html |
| wwwroot 未创建 | 静态文件未部署到正确位置 |
| 端口不一致 | 源代码用 5246，部署配置用 5000 |

### 问题2：启动后自动关闭

| 原因 | 说明 |
|------|------|
| 端口被占用 | 5000 端口被其他进程占用（如 VS Code 的 IIS Express） |
| wwwroot 路径错误 | WebRoot 未正确配置时，静态文件服务失败可能导致静默退出 |

### 问题3：localhost 拒绝连接

| 原因 | 说明 |
|------|------|
| 后端未成功启动 | 因端口占用或配置错误启动失败 |
| 端口配置差异 | 源代码 debug 配置用 5246，publish 配置用 5000 |

## 四、如何避免这些问题

### 1. 打包前检查清单

```markdown
□ 前端已执行 `npm run build`
□ wwwroot 目录已创建
□ dist/ 内容已复制到 wwwroot/
□ 端口配置已确认（默认 5000）
□ 没有其他进程占用 5000 端口
```

### 2. 验证部署包

```bash
# 检查 wwwroot 是否存在且有内容
ls backend/wwwroot/
# 应显示：assets/  favicon.svg  icons.svg  index.html

# 检查后端能否正常启动
./MesScanner.Backend.exe
# 应显示：Now listening on: http://localhost:5000

# 验证静态文件服务
curl http://localhost:5000/
# 应返回 HTML 内容，而非纯文本
```

### 3. 端口冲突检测

```powershell
# Windows 查看端口占用
netstat -ano | findstr ":5000"

# 关闭占用进程
taskkill /F /PID <进程ID>
```

## 七、注意事项

1. **自包含exe无需安装 .NET Runtime**，但首次启动较慢（约10-20秒）
2. **防火墙设置**：如需远程访问，需开放 5000 端口
3. **配置文件优先级**：部署目录下的 `Config/` 优先于程序内置配置
4. **前端代理配置（重要）**：开发环境的 Vite Proxy 在打包后失效，必须由后端进行转发映射。

## 八、生产环境 MES 接口转发说明

### 1. 转发原理
打包后的前端请求会发送到后端的 5000 端口。后端 `Program.cs` 拦截以 `/mes-api` 和 `/mes-push` 开头的请求，并将其转发到真实的服务器。

### 2. 路由映射逻辑
目前已在 `Program.cs` 中实现的转发规则：

| 前端路径 | 转发目标 (172.25.57.144) | 路径处理 |
| :--- | :--- | :--- |
| `/mes-api/*` | `http://...:8076/*` | **移除** `/mes-api` 前缀 |
| `/mes-push/*` | `http://...:8072/*` | **移除** `/mes-push` 前缀 |

### 3. 程序交互日志（黑窗口显示）
为了方便排错，后端开启了详细日志：
- `[MES代理] 转发请求`: 显示 HTTP 方法、前端路径及最终转发的目标 URL。
- `[代理响应正文]`: 在控制台显示 MES 返回的前 512 个字符数据。
- `[代理成功/失败]`: 显示 HTTP 状态码及转发耗时。

### 4. 常见问题：前端“等待响应”
如果在转发 POST 请求时前端挂起，通常是因为代理返回时 `Content-Length` 报头与实际字节长度不匹配。
**修复方案**：后端必须在向客户端写入数据前，清除旧的 `Content-Length` 并重新设置为当前字节数组的真实长度，最后调用 `FlushAsync()`。

### 5. 如何修改 MES 服务器地址 (已优化)
**无需重新打包！** 现在系统支持完全配置化：
1. 打开 `Config/app_config.json` (或通过 UI 界面 `系统配置`)。
2. 直接修改对应的 API 接口字段为完整的绝对地址（如 `http://172.25.57.144:8076/api/...`）。
3. 保存后系统会通过后端 `/api/proxy` 自动完成转发。

## 九、 常见故障记录 (2026-04-22)

### 问题 1：前端报错 "Failed to fetch" (API 转发失效)
**故障现象**：
在 VS Code 源码调试时，前端页面可以打开，但接口请求失败报错 `Failed to fetch`，在 F12 控制台中看到请求直接发往了 MES 服务器地址而非后端代理。

**原因分析**：
1. **环境错位**：源码后端（`backend/MesScanner.Backend`）默认加载同级目录下的 `wwwroot`。
2. **版本滞后**：执行 `npm run build` 后，最新的代码只更新了根目录的 `dist/`，没有同步到源码后端的 `wwwroot`。
3. **逻辑缺失**：浏览器中运行的是未实现 `/api/proxy` 代理包装的旧代码，导致触发跨域拦截。

**解决方案**：
确保在调试前执行同步命令（或通过 VS Code 预设任务）：
```powershell
# 将最新编译产物同步到源码后端目录
cp -r dist/* backend/MesScanner.Backend/wwwroot/
```

### 问题 2：调试器报错 "Couldn't find a debug adapter for 'dotnet'"
**故障现象**：
在 VS Code 中按 F5 启动调试失败。

**原因分析**：
`launch.json` 中的 `type` 设置为 `dotnet`（这是 C# Dev Kit 专用），而本地环境可能只安装了基础的 **C# (OmniSharp)** 扩展，无法识别该类型。

**解决方案**：
将 `launch.json` 中的 `type` 改为更通用的 `coreclr`。
